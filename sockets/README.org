* File Naming Convention
:PROPERTIES:
:VISIBILITY: folded
:END:

Many of the programs in this directory are named according to the
conventions described below. A prefix identifies the socket domain in
which the program operates, and the socket type employed. The socket
domain is indicated by one or two letters:

- =u=  UNIX
- =i=   Internet (both v4 and v6)
- =i6=  Internet v6 only

The socket type is one of the following:

- =d=   datagram
- =s=   stream

A suffix indicates whether the program is a client or server:

- =cl= client
- =sv= server

Thus, =id_echo_sv.c= is a server program that uses datagram sockets in
the Internet domain.

- Figure 56.1

| Domain     | Communication | Communication between | Address format         | Address structure |
|            | performed     | applications          |                        |                   |

|            |               | via an IPv4 network   | 16-bit port number     |                   |

|            |               | via an IPv6 network   | 16-bit port number     |                   |

* 56.4 Generic socket address structure

The only purpose for the =struct sockaddr= type is to cast the various
domain-specific address structures to a single type for use as
arguments in the socket system calls. The sockaddr structure is
typically defined as follows:

#+begin_src C
  struct sockaddr {
    sa_family_t sa_family; /* Address family (AF_* constant) */
    char sa_data[14]; /* Socket address (size varies according
                         to socket domain, 14 is arbitrary) */
  };
#+end_src

* 57.1 Unix Domain Socket Addresses: =struct sockaddr_un=

In the UNIX domain, a socket address takes the form of a pathname, and
the domain-specific socket address structure is defined as follows:

#+begin_src C
  struct sockaddr_un {
    sa_family_t sun_family;      /* Always AF_UNIX */
    char sun_path[108];          /* Null-terminated socket pathname */
  };
#+end_src

- The prefix =sun_= in the fields of the =sockaddr_un= structure has
  nothing to do with Sun Microsystems; rather, it derives from =socket
  unix=.

/SUSv3/ doesn’t specify the size of the =sun_path= field. Early BSD
implementations used 108 and 104 bytes, and one contemporary
implementation (HP-UX 11) uses 92 bytes. Portable applications should
code to this lower value, and use =snprintf()= or =strncpy()= to avoid
buffer overruns when writing into this field.

-Reference: [[file:us_socket.c][Listing 57.1 us_socket]].

** Explanation
In the (hypothetical) case that the =SV_SOCK_PATH= pathname is longer
than =sizeof(addr.sun_path) - 1= bytes, the =remove()= would attempt
to remove that pathname, but the =bind()= call might nevertheless fail
because it attempts to bind to a truncated version of the pathname

#+begin_src C
  /* The HACK: */
  if (strlen(SV_SOCK_PATH) > sizeof(addr.sun_path) - 1)
    fatal("Server socket path too long: %s", SV_SOCK_PATH);

Since the server socket pathname string in this program is actually
far smaller than the =sun_path= field, this check is not in practice
necessary for this program. However, in the general case (where the
pathname string size may be arbitrary), it is better to have such a
check, and the program should follow this good practice. (The first,
white-space change is made simply to allow enough space on the printed
page to accommodate the second change.)

* 57.3 Datagram Sockets in the UNIX Domain

In the generic description of datagram sockets that we provided in
Section 56.6, we stated that communication using datagram sockets is
unreliable.  This is the case for datagrams transferred over a
network.

However, for UNIX domain sockets, datagram transmission is carried out
within the kernel, and is reliable.  *All messages are delivered in
order and unduplicated.*

** Functions Prototypes
#+begin_src C
ssize_t sendto (int SOCKET, const void *BUFFER, size_t SIZE,
          int FLAGS, struct sockaddr *ADDR, socklen_t LENGTH)
#+end_src

The ‘sendto’ function transmits the data in the =BUFFER= through the
socket =SOCKET= to the destination address specified by the =ADDR= and
=LENGTH= arguments.  The =SIZE= argument specifies the number of bytes
to be transmitted ([[info:libc#Sending Datagrams][libc#Sending Datagrams]]).

#+begin_src C
ssize_t recvfrom (int SOCKET, void *BUFFER, size_t SIZE,
          int FLAGS, struct sockaddr *ADDR, socklen_t *LENGTH_PTR)
#+end_src

The ‘recvfrom’ function reads through the socket =SOCKET= one packet
into the buffer =BUFFER=.  The =SIZE= argument specifies the maximum
number of bytes to be read ([[info:libc#Receiving Datagrams][libc#Receiving Datagrams]]).  The =ADDR= and
=LENGTH_PTR= arguments are used to return the address where the packet
came from. Set both of them to =NULL= if uninterested by data origin.

** Why the Client Must =bind()=
:PROPERTIES:
:CUSTOM_ID: why-the-client-must-bind
:END:
Originally author used =sfd= for “socket file descriptor” in both
client and server.  It seems confusing at first so you could make it
contextual with variables like =clfd= and =svfd=.  But there is good
reason for a simple naming like =sfd=: in the UNIX family mostly in
DGRAMS type there is no “passive” or “active” socket in a
communication context.

The reason your brain likely snagged on the client's =sfd= is because
of the =bind()= call.

In *Internet Domain (UDP)* sockets, a client usually doesn't call
=bind()=; the OS just picks a random port. But in *UNIX Domain
(Datagram)* sockets, if the client wants a response, it *must* create
its own entry in the file system (the =/tmp/ud_ucase_cl.PID= file).

1. *Client Binds:* "I am at path X."
2. *Client Sends:* "Here is data. By the way, I am at path X." (This is
   passed via the =claddr= struct in =recvfrom=).
3. *Server Receives:* The server's =recvfrom= fills in =claddr= with the
   client's path.
4. *Server Sends:* The server uses that path to send the response back.

* 57.8 Exercises
** Exercise 57.1
:PROPERTIES:
:CUSTOM_ID: exercise-57-1
:END:
In Section 57.3, we noted that UNIX domain datagram sockets are
reliable. Write programs to show that if a sender transmits datagrams
to a UNIX domain datagram socket faster than the receiver reads them,
then the sender is eventually blocked, and remains blocked until the
receiver reads some of the pending datagrams.

*** What the exercise is all about
The goal is simply to prove that *UNIX Datagrams act like a pipe with
a fixed diameter*: if you pour water in faster than it can come out
the other side, the water eventually backs up to the source.

You need to create *congestion* on the receiver's kernel buffer by
forcing it to sleep while the sender is flooding it with thousands of
datagram bytes sent in a tight loop.  When the receiver's kernel
buffer get saturated the kernel will turn back to the sender and block
the send() syscall forcefully pausing the process at that very code
line until the receiver start making space in its buffer by unclogging
its waiting queue.

To have clear result limit the test lab to unidirectional I/O.

In a peer-to-peer relationship like UNIX domain datagrams, "Server"
and "Client" are just labels. The *Blocking/Flow Control* logic
applies to any sender attempting to write to a full socket buffer,
regardless of their role.

You can check the actual size of the buffer on your Linux machine by
looking at =/proc/sys/net/core/rmem_default= with some pager or simply
the =cat= command, or by using =getsockopt()= with
=SO_RCVBUF=. Usually, it's around *212 KB*. If your messages are 1 KB,
the client will block after about 200 messages.

*** Answer
**** How to shift the logic (The "Sender-Side" Bottleneck)
:PROPERTIES:
:CUSTOM_ID: how-to-shift-the-logic-the-sender-side-bottleneck
:END:
To set the client as the “flooder” it is enough to send massive amount
of datagrams in a tight loop while the server sleeps *before* entering
its infinite loop.

In this scenario, the *Server* becomes the high-speed sender (the
"flooder") and the *Client* becomes the slow consumer (the
"bottleneck").

1. *The Client (Receiver):*
   - Sends an initial "Trigger" datagram to the server so the server
     knows the client's address (=claddr=).
   - Immediately goes to =sleep(30)=.
   - During this sleep, its receive buffer will fill up.
2. *The Server (Sender):*
   - Receives the trigger.
   - Enters a =for= loop to send thousands of responses back to that
     =claddr= as fast as possible.
   - *The Observation:* The Server will eventually hang on =sendto()=.

*** Technical Summary
**** The Logic of the "Blocking" Point
:PROPERTIES:
:CUSTOM_ID: the-logic-of-the-blocking-point
:END:
You don't need the =read()= and =write()= handshake because the kernel
manages the socket buffer independently of the application logic.

1. *Phase 1 (The Pour):* The Sender pours datagrams into the socket.
2. *Phase 2 (The Fill):* The Kernel places those datagrams into the
   Receiver's *Receive Queue (Recv-Q)*.
3. *Phase 3 (The Block):* Once the Recv-Q reaches its byte limit
   (defined by the kernel), the next =sendto()= call cannot move the
   data from the Sender's process into the kernel.
4. *Result:* The =sendto()= system call *blocks*. It effectively
   "pauses" the Sender's execution right at that line of code.

**** What You Will Observe
:PROPERTIES:
:CUSTOM_ID: what-you-will-observe
:END:
If you run the "Slow" Server and the "Fast" Client: 1. The Client will
print "Sending message #1", "#2"... very quickly. 2. Suddenly, the
Client will *stop* printing and hang on a =sendto()= call. This is the
moment the kernel's socket buffer is full. 3. Because UNIX domain
sockets are *reliable*, the kernel won't just drop the packets (like
UDP might). It pauses the Client process. 4. Once the Server's
=sleep(30)= ends and it calls =recvfrom()=, the Client will suddenly
wake up and continue sending more messages.

**** Why this proves the point
:PROPERTIES:
:CUSTOM_ID: why-this-proves-the-point
:END:
Whether it's the Server or the Client that is sleeping, the core
mechanics of *AF_UNIX* remain the same:

- *Reliability:* Unlike *AF_INET (UDP)*, which would simply drop the
  packets (silent loss), the UNIX domain protocol ensures every byte
  is delivered.
- *Backpressure:* When the receiver's queue is full, the kernel
  applies "backpressure" to the sender. The sender's =sendto()= call
  cannot complete until the receiver's kernel buffer has enough free
  space to hold the new datagram.

**** A Practical Experiment for your Script
:PROPERTIES:
:CUSTOM_ID: a-practical-experiment-for-your-script
:END:
If you want to see the exact moment the block happens without waiting
for a full 30-second sleep, you can run your "flooder" in one terminal
and use the =ss= (socket statistics) command in another:

#+begin_src sh
  # Run this while your sender is active
  ss -axp | grep ud_ucase
  # or
  ss -a --unix | grep ud_ucase
#+end_src

Look for the *Recv-Q* and *Send-Q* columns. * *Recv-Q:* You will see
this number climb until it hits the max limit of the receiving
socket. * *The Hang:* The moment *Recv-Q* hits the ceiling, your
sender's process state will switch to =D= (uninterruptible sleep) or
=S= (interruptible sleep) while it waits for the buffer to clear.
