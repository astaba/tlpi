*** Exercises 29.1
What possible outcomes might there be if a thread executes the
following code:

#+begin_src C
pthread_join(pthread_self(), NULL);
#+end_src

Write a program to see what actually happens on Linux. If we have a
variable, tid, containing a thread ID, how can a thread prevent itself
from making a call, pthread_join(tid, NULL), that is equivalent to the
above statement?

**** Answer
:PROPERTIES:
:VISIBILITY: folded
:END:
Reference:
- [[file:exr2901.c][Exercise 29.1 exr2901]].

Clean code and good exercise. Your commented block is particularly
valuable --- you ran the naive case, got ~EDEADLK~, and recorded it
right there in the source. That's good experimental discipline.

A few observations:

***** On the deadlock result
Linux actually returns ~EDEADLK~ immediately rather than truly
deadlocking --- it detects the self-join and refuses it. POSIX leaves
this behavior undefined, so Kerrisk's question "what possible outcomes"
is genuinely open. On some implementations it could hang forever. Linux
being defensive here is a quality-of-life choice, not a guarantee.

***** On your guard
#+begin_src C
if (!pthread_equal(tid1, pthread_self()))
#+end_src

This correctly answers the second part of the exercise. Using
~pthread_equal~ rather than ~tid1 != pthread_self()~ is the right call
--- POSIX does not guarantee ~pthread_t~ is a scalar type that supports
~==~ directly. On Linux it happens to be an unsigned long, but your code
is portable, which matters.

***** One thing worth thinking about
Your guard protects ~main~ from joining itself, but consider --- could
~tid1~ ever equal ~pthread_self()~ in ~main~ here? The newly created
thread gets a fresh tid, so practically no. But the exercise is really
teaching you to build the habit of the check for cases where thread
IDs are passed around as arguments and a thread cannot know in advance
whether the tid it received is its own.

*** Exercise 29.2
Aside from the absence of error checking and various variable and structure
declarations, what is the problem with the following program?

#+begin_src C
  #include <pthread.h>

  static void *

  threadFunc(void *arg) {
    struct someStruct *pbuf = (struct someStruct *) arg;
  /* Do some work with structure pointed to by 'pbuf' */
  }

  int main(int argc, char *argv[]) {
    struct someStruct buf;
    pthread_create(&thr, NULL, threadFunc, (void *) &buf);
    pthread_exit(NULL);
  }
#+end_src

**** Answer: The problem is *stack frame lifetime*.
:PROPERTIES:
:VISIBILITY: folded
:END:

=buf= is a local variable of =main=. It lives on =main='s stack frame.
When =pthread_exit(NULL)= is called, main's stack frame is destroyed
--- but the peer thread =threadFunc= received a pointer to that
now-dead memory and may still be working with it.

So =pbuf= becomes a *dangling pointer* the moment main exits, even
though the process continues because =pthread_exit= only terminates
the calling thread, not the process itself. That's the subtlety
Kerrisk is testing --- most people's instinct is that =pthread_exit=
is "safe" here precisely because it doesn't kill the process. And it
is safe for the process. But not for the stack.

The race is:

#+begin_example
main creates thread → main calls pthread_exit → main's stack destroyed
                                    ↕
        threadFunc still reading buf via pbuf → undefined behavior
#+end_example

**** The fixes in increasing robustness
:PROPERTIES:
:VISIBILITY: folded
:END:

***** 1. Heap allocate
#+begin_src C
struct someStruct *buf = malloc(sizeof(*buf));
pthread_create(&thr, NULL, threadFunc, buf);
// thread is responsible for free()
#+end_src

***** 2. Join before exiting
#+begin_src C
pthread_create(&thr, NULL, threadFunc, &buf);
pthread_join(thr, NULL);
// buf guaranteed alive until thread finishes
#+end_src

***** 3. Global or static
#+begin_src C
static struct someStruct buf;
// Lifetime extends beyond main's frame.
#+end_src

**** Summary
:PROPERTIES:
:VISIBILITY: folded
:END:
The deeper lesson Kerrisk is planting here is that *stack and thread
lifetime are orthogonal*. Every previous chapter let you assume a
stack frame lives exactly as long as its function. Threads break that
assumption completely and permanently.

** 30.2 Signaling Changes of State: Condition Variables

- [[file:prod_no_condvar.c][Suplprog 30.2.A prod_no_condvar]]: A simple POSIX threads
  producer-consumer example that doesn't use a condition variable.

The ~units~ consumer loop (~for (;;)~) in the main routine
works, but it wastes CPU time, because the main thread continually
loops, checking the state of the variable avail. A condition variable
remedies this problem. It allows a thread to sleep (wait) until
another thread notifies (signals) it that it must do something (i.e.,
that some “condition” has arisen that the sleeper must now respond
to).

- [[file:prod_condvar.c][Suplprog 30.2.B prod_condvar]]: A simple POSIX threads
  producer-consumer example using a condition variable.

A condition variable is always used in conjunction with a mutex. The
mutex provides mutual exclusion for accessing the shared variable,
while the condition variable is used to signal changes in the
variable’s state. (The use of the term signal here has nothing to do
with the signals described in Chapters 20 to 22; rather, it is used in
the sense of indicate.)

*** 30.2.4 Example Program: Joining Any Terminated Thread
- [[file:thread_multijoin.c][Listing 30.4 thread_multijoin]]: A main thread that can join with any
  terminated thread.

Finally, note that although the threads in the example program are
created as joinable and are immediately reaped on termination using
=pthread_join()=, we don’t need to use this approach in order to find
out about thread termination. We could have made the threads detached,
removed the use of =pthread_join()=, and simply used the thread array
(and associated global variables) as the means of recording the
termination of each thread.

**** Main thread =totThreads= and =numLive= global variables
:PROPERTIES:
:CUSTOM_ID: main-thread-totthreads-and-numlive-global-variables
:END:

These variables are not shared but only ever used by the main thread.

- =totThreads=: Keep the total number of threads created. Required
  whithin the zombie-threads joiner-loop of the main thread to loop
  the entire thread-monitor array. The reason is thread termination is
  completely unrelated with their index order. That is why the
  thread-monitor-array walking-loop (within the zombie-threads
  joiner-loop) compares idx (its iterator) to totThreads and not to
  numLive.

- =numLive=: Decrementally updates the total number of threads
  created. Required in the main thread to keep track of reaped threads
  and providing the zombie-threads joiner-loop with a terminating
  condition.

**** Why are they global if only ever used by main thread?
:PROPERTIES:
:CUSTOM_ID: why-are-they-global-if-only-ever-used-by-main-thread
:END:

#+begin_quote
/Is it pedagogical to foster good practice for real scenarios where it
is more often the case?/
#+end_quote

It's partly pedagogical but there's a more concrete reason too. They
live alongside the shared variables =numUnjoined= and =thread= which
genuinely need global scope. Keeping all program-state variables at
the same level is a *cohesion* choice --- the alternative of passing
them down as locals or parameters would fragment the mental model of
"here is the complete state of this program." In a real server these
would likely become fields of a single program-state struct anyway.

**** Real Point of Mutual Exclusion
:PROPERTIES:
:CUSTOM_ID: real-point-of-mutual-exclusion
:END:

Why is =thread[idx].sleepTime= accessed without a lock in
=threadFunc=?

#+begin_quote
/Is it because these specific memory slots are "constant" and
"private" to every single thread, and independent from any shared
variables?/
#+end_quote

Your attempted answer is exactly right and precisely stated. The key
insight is *write-once before thread creation, read-only after*. Main
writes =sleepTime= and =state= for each slot before calling
=pthread_create=. Once the thread is live, it only reads its own
=sleepTime= --- no other thread touches that slot's =sleepTime= ever
again. There is no concurrent write anywhere in the program's lifetime
for that field, so no protection needed.

***** Mutex Important General Principle
:PROPERTIES:
:CUSTOM_ID: mutex-important-general-principle
:END:

*Mutual exclusion protects concurrent writes or concurrent write and
read, not reads of immutable data.*

**** Index transmission via cast is contrived
:PROPERTIES:
:CUSTOM_ID: index-transmission-via-cast-is-contrived
:END:

How to pass additional arguments orthodoxly?

You're right that =(void *)idx= is a hack that only works because an int
fits in a pointer on the target platform --- which POSIX does not
formally guarantee. The orthodox solution when you need richer
per-thread arguments is exactly the struct-pointer pattern:

#+begin_src C
typedef struct {
    int idx;
    int someOtherArg;
    char *message;
} ThreadArg;

ThreadArg *arg = malloc(sizeof(*arg));
arg->idx = idx;
arg->someOtherArg = 42;
pthread_create(&tid, NULL, threadFunc, arg);
// thread is responsible for free()
#+end_src

Each thread gets its own heap-allocated argument struct. Kerrisk uses
the cast shortcut here precisely because the exercise is about joining
and state management, not argument passing --- he doesn't want the
boilerplate to obscure the point.

**** No malloc cleanup
:PROPERTIES:
:CUSTOM_ID: no-malloc-cleanup
:END:
Why no =free(thread)=?

#+begin_quote
/Perhaps because at this point (instruction pointer) it doesn't matter
for the pedagogical purposes./
#+end_quote

Correct on both counts --- pedagogical shortcut and timing. But there's
a sharper reason worth adding to your note: =exit()= is called, not
=return=. =exit= tears down the process immediately --- all heap memory
is reclaimed by the OS, all file descriptors closed. So the leak is
genuinely inconsequential here. In a library function or a server that
doesn't exit, omitting =free= would be indefensible.

The real discipline is: =free= everything that =malloc=ed it, unless
process termination is guaranteed and imminent. Kerrisk knows exactly
what he's doing here --- the absence of =free= is a conscious choice,
not an oversight.

**** Summary
All four of your readings were sound. The third one about index
transmission shows particularly sharp instinct --- that cast is one of
those things that works on Linux x86-64 today and is technically
nonportable, and recognizing it as contrived is exactly right.
